# FIFA 2026 — 賠率與貝葉斯分析說明

> 本文件說明 `fifa2026_schedule.html` 中「賠率(先驗)」與「貝葉斯後驗」兩者的實際實作、數學邏輯,以及目前程式碼與交接文件 `fifa2026_dev_handoff.md` 之間的差異。
> 對應原始碼:`oddsToProb()`、`bayesUpdate()`、`buildTeamProfiles()`(約 376–458 行)。

---

## 一、整體流程

```
外盤賠率 ──oddsToProb()──▶ 先驗概率(藍條) ──bayesUpdate()──▶ 後驗概率(紫條)
                                              ▲
                                  buildTeamProfiles() 各隊本屆表現
```

- **藍條** = 市場賠率去抽水後的概率(客觀,但滯後)。
- **紫條** = 在藍條基礎上,疊加本屆各隊真實戰績做的修正。
- **↑↓ 箭頭** = 兩者差距 > 1.5% 時顯示,即修正方向的可視化。

---

## 二、第一步:賠率 → 先驗概率(藍條)

`oddsToProb(home, draw, away)`,功能是 **去除莊家抽水(vig)**:

```js
raw   = 1 / 賠率              // 賠率倒數 = 隱含概率(含水)
total = raw_h + raw_d + raw_a  // 通常 > 1,多出來的就是莊家抽水
概率  = raw / total * 100      // 正規化回 100%
```

這一步**不含任何「分析」**,只是把市場賠率翻譯成乾淨的概率。

---

## 三、第二步:貝葉斯後驗(紫條)

公式骨架是標準貝葉斯形式:

```
後驗 ∝ 先驗 × 似然

unnorm_h = 先驗_主勝 × L(主勝)
unnorm_d = 先驗_平   × L(平)
unnorm_a = 先驗_客勝 × L(客勝)
posterior = normalize(unnorm_h, unnorm_d, unnorm_a)
```

其中每種結果的似然由雙方表現相乘:

```js
L(主勝) = perfFactor(主隊,'win')  × perfFactor(客隊,'loss')
L(平)   = perfFactor(主隊,'draw') × perfFactor(客隊,'draw')
L(客勝) = perfFactor(客隊,'win')  × perfFactor(主隊,'loss')
```

### perfFactor — 核心啟發式打分

> 數值 > 1 代表該結果比賠率預期更可能發生;< 1 則相反;球隊 0 場 → 回傳 1(不修正)。

**win** 的算法:
```js
1 + (勝率 - 0.4) * 1.2 + 場均淨球 * 0.08 + 爆冷次數 * 0.15 - 被爆次數 * 0.12
// 並套用 Math.max(0.4, …) 下限
```

| 項目 | 效果 |
|------|------|
| `(勝率-0.4)*1.2` | 勝率高於 40% 加分,低於扣分 |
| `場均淨球*0.08` | 進球多、失球少加分 |
| `爆冷次數*0.15` | 製造過冷門 → 視為黑馬,加分 |
| `被爆次數*0.12` | 被爆冷過 → 扣分 |
| `Math.max(0.4, …)` | 下限 0.4,不會歸零 |

**draw**:`Math.max(0.5, 1 + (平局率 - 0.25) * 0.8)`
**loss**:`Math.max(0.4, 1 + (敗率 - 0.3) * 0.9 - 被爆次數*0.12)`

## 四、目前實作與文案的差異(容易困惑的點)

1. **「先驗」有兩個來源,程式只用其一**
   - 即將開賽場次:先驗來自**即時賠率**(`oddsCellHtml` 呼叫 `oddsToProb`)。
   - 歷史場次物件雖存有 `ph/pd/pa`(賽前先驗),但目前藍/紫條**只渲染在 scheduled/upcoming 場次**,歷史頁不顯示貝葉斯。因此 `ph/pd/pa` 現階段幾乎沒被使用。

2. **`buildTeamProfiles` 的 `expected_wins` 欄位算了卻沒用** — 預留欄位,後續無人讀取。

3. **爆冷歸屬依賴 `g.ph`** — 只有歷史場次有填 `ph` 才算得出誰製造冷門;未填 `ph` 的爆冷場次,`upsets_caused / upset_suffered` 不會累加,perfFactor 的爆冷項即失效。

4. **這並非嚴格的貝葉斯** — `perfFactor` 不是真正的條件機率 P(資料|假設),而是**手調係數的啟發式打分函數**(1.2 / 0.08 / 0.15… 皆人工設定)。本質上是「用本屆戰績對市場賠率做加權微調」,披著貝葉斯外衣。

---

## 五、一句話總結

- **藍條(賠率)** = 市場怎麼看,純去抽水,客觀但滯後。
- **紫條(貝葉斯)** = 在市場看法上疊加本屆各隊真實戰績的修正:戰績好/有黑馬表現的隊概率被往上推,反之往下。箭頭 ↑↓ 即這層修正的可視化。

## 六、後續可改進方向

1. **實算驗證** — 將目前 `ALL_GAMES` 代入,輸出某幾場的藍條 vs 紫條數字,直觀檢視修正幅度。
2. **改用正規統計模型** — 以 Poisson / Elo / Dixon-Coles 取代啟發式打分,賠率當先驗、戰績當似然。
3. **修正不一致點** — 讓歷史頁也顯示貝葉斯、啟用 `expected_wins`、補齊爆冷場次的 `ph` 欄位。
