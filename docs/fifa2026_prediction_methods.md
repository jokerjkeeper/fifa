## 一、目前的設計（What we have now）

### 整體流程

```
外盤賠率 ──oddsToProb()──▶ 先驗機率(藍條) ──bayesUpdate()──▶ 後驗機率(紫條)
                                              ▲
                                  buildTeamProfiles() 各隊本屆戰績
```

- **藍條 = 市場賠率去抽水後的機率**：客觀，但滯後（賽前定盤後不動）。
- **紫條 = 在藍條上疊加本屆各隊真實戰績的修正**。
- **↑↓ 箭頭**：兩者差距 > 1.5% 時顯示，是修正方向的可視化。

### 第一步：賠率 → 先驗（`oddsToProb`）

純粹去除莊家抽水（vig）：

```js
raw   = 1 / 賠率              // 賠率倒數 = 含水隱含機率
total = raw_h + raw_d + raw_a // 通常 > 1，多出來的是抽水
機率  = raw / total * 100     // 正規化回 100%
```

這一步**不含任何分析**，只是把市場賠率翻譯成乾淨的機率。這部分是正確且應保留的。

### 第二步：貝葉斯後驗（`bayesUpdate`）

標準貝葉斯骨架 `後驗 ∝ 先驗 × 似然`：

```
L(主勝) = perfFactor(主隊,'win')  × perfFactor(客隊,'loss')
L(平)   = perfFactor(主隊,'draw') × perfFactor(客隊,'draw')
L(客勝) = perfFactor(客隊,'win')  × perfFactor(主隊,'loss')
posterior = normalize(先驗 × L)
```

核心 `perfFactor` 是**手調係數的啟發式打分**，例如 win：

```js
1 + (勝率-0.4)*1.2 + 場均淨球*0.08 + 爆冷次數*0.15 - 被爆次數*0.12   // 下限 0.4
```

| 項目 | 效果 |
|------|------|
| `(勝率-0.4)*1.2` | 勝率高於 40% 加分 |
| `場均淨球*0.08` | 進球多、失球少加分 |
| `爆冷次數*0.15` | 製造冷門 → 視為黑馬，加分 |
| `被爆次數*0.12` | 被爆冷過 → 扣分 |

`draw = max(0.5, 1+(平局率-0.25)*0.8)`；`loss = max(0.4, 1+(敗率-0.3)*0.9 - 被爆*0.12)`。

## 二、目前設計的根本問題（Why it's not "real" Bayes）

1. **不是真正的條件機率。** `perfFactor` 不是 `P(資料 | 假設)`，而是一個值域大致在 0.4~2 的人工打分函數。`1.2 / 0.08 / 0.15` 全是拍腦袋的係數，沒有可驗證的機率意義。本質是「**用本屆戰績對市場賠率做加權微調**」，披著貝葉斯外衣。

2. **似然無法被資料校準。** 因為係數是寫死的，沒辦法用實際比賽結果去擬合 / 反推，也就無法回答「這個修正到底有沒有讓預測更準」。

3. **資料缺口導致部分修正失效。**
   - 爆冷歸屬依賴歷史場次的 `ph` 欄位；未填 `ph` 的場次 `upsets_caused / upset_suffered` 不會累加，爆冷項即失效（目前 40+ 場 final 只有 ~8 場有 `ph`）。
   - `expected_wins` 算了沒被讀取。

4. **時間 / 場次洩漏風險（leakage）。** `buildTeamProfiles()` 用「全部 final 場次」建檔。若拿來「驗證」歷史某場的預測，會用到那場之後才發生的比賽，造成未來資訊洩漏，驗證結果會虛高。正確做法是只用「該場開賽前」已完成的比賽建檔。

5. **歷史頁不顯示賠率 / 機率。** 目前只有 upcoming / today 渲染藍紫條，history 頁沒有，導致無法回頭比對「預測 vs 實際」。（本次已著手補上顯示，但受限於歷史賠率未被記錄，見第四節。）

## 三、更好的方法（Alternatives，按推薦度排序）

足球預測的主流工具其實都落在機率論的既有版圖內（對照 `隨機.md`）：

### ⭐ 首選：Poisson / Dixon-Coles 模型
- **原理**：足球進球數近似**泊松分布**。估每隊的「進攻強度 λ_attack」「防守強度 λ_defense」，算出主客隊各進 0,1,2… 球的機率，外積成比分矩陣，再把矩陣加總成勝/平/負。
- **Dixon-Coles** 是業界經典改良：修正低比分（0-0,1-0,0-1,1-1）的相關性，並加**時間衰減權重**（近期比賽權重高）。
- **對本專案的契合點**：可「**賠率當先驗、進球數當似然**」，這才是名副其實的貝葉斯，而且每個參數都有機率意義，能用對數似然 / Brier score 校準與驗證。能直接輸出**精確比分**機率。
- **成本**：純前端 JS 可實作，無需後端。中等工作量。

### ⭐ Elo / Glicko 評分
- **原理**：每隊一個動態實力分，賽後依「實際 vs 預期」更新。World Football Elo、FIFA 官方排名都是這套。
- **優點**：極簡、只需勝負結果、天然把「實力差 → 勝率」映射好。
- **用法**：可單獨給先驗，或作為 Poisson 的 λ 輸入。Glicko 額外帶「評分不確定度」。

### Bradley-Terry / Ordered Logit
- 專做成對比較與**有序三分類（負<平<勝）**的回歸，用極大似然估計（MLE）擬合。比現在的打分函數嚴謹，仍輕量、純前端可算。

### 機器學習（需特徵 + 後端）
- 有 xG、陣容、傷停、主客場等特徵時，用 **XGBoost / 邏輯回歸**做三分類。準度上限最高，但需要訓練資料與後端，對一個靜態 HTML 看板偏重。

### 可借鑑的進階工具（對照 `隨機.md`）
- **最大熵原理 / KL 散度**：在滿足賠率約束下找「最不武斷」的分布；用 KL 衡量後驗與市場的偏離。
- **層級貝葉斯 + MCMC**：把球隊強度當隱變量、設先驗分布，估出帶**不確定區間**的預測。這才是貝葉斯學派的完整形態。
- **Brier score / log-loss**：標準的機率預測評分，用來客觀回答「我們有沒有勝過裸賠率」。

## 四、本專案的演進建議

| 方案 | 工作量 | 需後端 | 收益 |
|------|--------|--------|------|
| **記錄每場賽前賠率（資料底座）** | 小 | 否 | ⭐⭐⭐ 沒有它，後面所有驗證都做不了 |
| 歷史頁顯示「賠率→貝葉斯 vs 實際結果」 | 小 | 否 | ⭐⭐ 肉眼比對、揭露資料缺口 |
| Brier / log-loss 回測（裸賠率 vs 貝葉斯） | 小 | 否 | ⭐⭐⭐ 驗證修正是否真的有用 |
| **Poisson + 賠率先驗**（取代 perfFactor） | 中 | 否 | ⭐⭐⭐ 真·貝葉斯、能出比分、可校準 |
| Elo 評分餵 λ | 中 | 否 | ⭐⭐ 動態實力、簡潔 |
| 修正 leakage（只用賽前資料建檔） | 中 | 否 | ⭐⭐ 讓驗證數字可信 |
| XGBoost 三分類 | 大 | 是 | ⭐ 上限最高，偏離當前架構 |

### 推薦路線

1. **先建資料底座**：每場比賽記錄「賽前賠率快照」（或至少 `ph/pd/pa` 先驗），這是驗證一切的前提。
2. **歷史頁可視化 + Brier 回測**：先用既有資料量化「目前 heuristic 到底準不準」。
3. **替換引擎**：把 `perfFactor` 換成 Poisson/Dixon-Coles 似然，並修掉 leakage（建檔只用賽前場次）。
4. 之後若有餘力再上 Elo 餵 λ、層級貝葉斯。

> 一句話：保留「賠率去抽水當先驗」這個正確起點，把披著貝葉斯外衣的 `perfFactor` 換成可校準的 Poisson 似然，並補上「記錄賽前賠率 + Brier 回測」這條驗證閉環。
